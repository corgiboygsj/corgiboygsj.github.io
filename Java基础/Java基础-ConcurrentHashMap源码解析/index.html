之前我们说过HashMap无论是1.7还是1.8，在并发情况下都会出现各种各样的问题。虽然JDK中提供了同步集合包装器和HashTable，但是他们都是简单的对方法进行上锁，锁的粒度很粗，因此他们在高并发下的性能都不会很好。本文章都是围绕JDK1.8来说的。

Java中如何使用线程安全的Map

HashTable

  • HashTable不支持null的key和value。
  • HashTable只是简单的使用key的hash值和数组长度做取余计算下标。
  • HashTable没有红黑树的结构转换，只有是数组和链表组成的散列表结构。
  • HashTable的同步只是对方法进行synchronized加锁。

同步包装器

Collections类下提供了同步的集合包装器方法。

我们看源码可以发现，这里使用了装饰器模式。将传进来的的对象通过SynchronizedMap内部类进行，在SynchronizedMap中有一个锁，在每个方法中调用了传进来的对象的方法，同时在调用前后进行加锁。这样的锁粒度也是整个对象级别的。

ConcurrentHashMap

ConcurrentHashMap解决了前面两种锁粒度过粗引起的并发度较低的问题。因为我们知道HashMap是一种散列表的结构，自然就可以发现不同桶之间数据是不存在竞争的，所以像前面那两种方式就会使得不同桶之间的操作也是互斥的。如果可以把同步的粒度调整到以桶为单位就可以极大的提高并发度。ConcurrentHashMap通过cas操作和对桶头的加锁将同步粒度控制在了以桶为单位，而且在哈希表扩容的时候ConcurrentHashMap还可以充分利用多线程的优势，使用多个线程来同时进行扩容。

put方法

总结一下，整体来说put方法通过无锁cas替代加锁尽可能的优化了性能，如果桶不为空情况则通过对桶头元素加锁实现按桶粒度的同步。这里之所以不再使用lock而是直接使用synchronized是因为在jdk升级过程中对synchronized进行了优化，具体优化了什么可以看另一篇博文Synchronized关键字原理。

代码注意点2：UnSafe操作数组

接下来看下代码注意点2，这三个方法都是存在于ConcurrentHashMap中的，对于tabAt和setTabAt我们可能想问为什么不能直接tab[i]=obj进行赋值，而要使用UnSafe提供的方法。

这里涉及到了volatile的原理，我们知道volatile可以保证内存的可见性。但是对于volatile的数组而言，只有这个数组的引用是保证内存可见性的，但是内部的元素是不保证的。也就是说例如一个volatile数组table，table = new int[]{11};赋值后会被立刻刷入内存中保证可见性，但是table[0] = 1;这样就不会被立刻强制刷入内存，因为volatile是不保证数组内元素可见性的。所以tabAt和setTabAt使用UnSafe提供的方法进行操作就是为了解决数组内元素的内存可见性问题。

size方法

因为concurrentHashMap是在并发情况下使用的，那么在计算size的同时也可能有别的线程在插入数据或者删除数据引起size的改变。

那么现在重点是看看这个baseCount的值和CountCell的数组中的值是如何记录的。可以看到put方法的最后会调用一个addCount(1L, binCount);的方法。

总结一下

size的计算是通过baseCount的值和counterCell数组中所有的值计算一个总和。
在put的过程中，调用了addCount方法，首先是修改baseCount的值，一旦发生竞争某个线程的baseCount修改失败，则会生成随机数通过散列获取countercell数组中的一个并修改值，如果失败就会通过自旋设置countercell的值直到成功为止，自旋内部如果还是竞争就会将countercell扩容为二倍，重新生成随机数进行尝试。
我认为countercell就是用来记录并发修改baseCount失败的值的，之所以在自旋修改countercell时候再失败会扩容也是为了减少线程冲突的概率，从而提升代码运行的速度，使用空间换时间。

扩容方法

扩容的大致流程如下，详细的在代码中有注释：
1、通过cpu核心数量计算一个线程应该负责的桶数量，每个线程最少处理桶的数量是16.
2、之后线程会来通过cas给总桶的数量减去任务桶数量，也就是领取任务。
3、线程会锁住桶的头节点防止并发插入新的键值对，在完成桶的扩容后会在把头节点变成占位符表示桶处理结束了，其他的线程如果遇到桶的头节点是占位节点就会跳过。
4、当线程处理完成后会回来继续领取任务。
5、全部都结束之后会重新检查一遍数组是否有遗漏。

如果put时数组位置有这个占位节点就表示正在进行扩容，而且这个桶已经完成了，那么干等着不如来帮助进行扩容。之后helpTransfer内部也会调用transfer方法让这个线程领取任务帮助完成扩容。

总结一下

ConcurrentHashMap的扩容方法设计的绝对是精华、也很牛逼。扩容都使用了多线程的方式来实现，它通过给线程分配自己的任务区间的方式让线程之间的扩容操作不互相影响，充分提高速度。通过给头节点加锁防止扩容期间有put操作影响链表数据，put的时候还会让当前线程来帮助进行扩容而不是阻塞。
