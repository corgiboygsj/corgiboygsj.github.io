AbstractQueuedSynchronizer源码分析

本文详细解读了AQS框架控制并发的原理，以及基于AQS怎么实现不同的锁

Java并发包中的锁工具都是基于AQS来实现的。AQS是并发包的核心，它是一个加强版的CLH队列，它实现了线程阻塞唤醒、取消节点、支持重入、支持独占、共享锁等多种基础功能，其他并发工具直接基于AQS实现自己的特殊的功能即可。

什么是CLH队列

CLH队列是一个先进先出队列，本质是通过队列与节点的状态控制节点的阻塞与唤醒。其中每一个线程代表队列中的一个节点，它不断自旋监听前驱节点的状态（前驱节点是否释放了锁）。前驱节点释放锁之后，后继结束自旋，改变自己的状态进行锁的获取。

Java中AQS是怎么使用的

首先AQS全名是AbstractQueuedSynchronizer，它是一个抽象类，相比于CLH队列，它不是通过一直自旋监听前驱的状态，而是通过前驱状态选择自己是否进行阻塞，前驱节点状态改变时选择是否唤醒后继节点。而且实现了很多CLH不具备的功能。
它基于CLH实现了共享锁、独占锁的基本功能（节点插入，节点阻塞、节点唤醒逻辑），没有实现的只是尝试获取锁的操作，因为尝试获取锁能不能成功、以什么样的逻辑成功决定了锁的特性，AQS只是个架子，比如ReentranLock、ReentrantReadWriteLock、CountDownLatch等都是通过实现了不同的尝试获取锁方法来实现自己的不同的特性的。
公平锁与非公平锁也只是尝试获取锁时的逻辑不同，非公平锁可以尝试获取锁，而公平锁就需要老老实实排队而已。

AQS源码分析

重要的属性

AQS的重要属性。

其中state是一个非常重要的属性，它代表了锁是否被占用的状态，同时还代表锁的占用次数，比如重入锁的重入次数。当然也可以基于state字段实现其他的功能，比如CountDownLatch就是通过state计数表示还有多少线程没有调用countDown方法，从而让使用await方法的线程阻塞。

队列节点类Node的重要属性。

其中比较重要的是节点的四个属性，它们虽然代表自己的状态但是也会影响其他节点的状态，比如SIGNAL状态就代表自己后面的线程需要被阻塞。

添加节点

首先先看AQS是如何把节点放到队列中的

节点插入流程：

  • 判断队列是否为空，如果为空则CAS将头指向自己。
  • 不为空则自旋不断通过CAS尝试将节点插入到队尾。

独占锁

这里是以ReentrantLock类为例，说一下非公平独占锁是怎么实现的。

获取锁（lock方法）

获取锁的流程：

  • 进入方法就先尝试看看state是不是0，如果是代表没有人占用就将其设置成1，之后将独占锁的线程指向自己表示被自己占用了。
  • 如果state不是0表示已经有人获取了锁，那么就要进行acquire方法进行锁的获取。

这个方法主要分为tryAcquire和acquireQueued方法，tryAcquire是一个尝试获取锁的方法（ReentranLock自己实现的），如果获取失败了，则调用addWaiter将自己入队，通过acquireQueued执行线程阻塞逻辑以及唤醒后争抢锁的逻辑。addWaiter方法的实现在前面写过了。

首先看一下tryAcquire方法。

tryAcquire方法的流程为：

  • 判断一下锁是否被占用了，如果没有占用则自己来占用。
  • 如果被占用了的话，判断一下锁的持有者是不是自己，如果是自己代表自己又调用了lock方法，就要进行锁的重入，就将state加一。

tryAcquire方法获取锁失败了的话，就执行acquireQueued(addWaiter(Node.EXCLUSIVE), arg))。将线程加入队列，之后阻塞线程，等待持有锁的线程释放锁并唤醒自己，下面是acquireQueued的实现。

acquireQueued方法的流程为：

  • 看看自己的前驱是不是头节点，如果是的话就有可能前驱释放了锁，那么自己进行获取锁的尝试。
  • 尝试失败的话，则判断前驱节点的状态是不是SIGNAL，SIGNAL表示了后继需要被阻塞，如果是话就调用Unsafe的线程阻塞方法阻塞自己，之后线程就会阻塞在parkAndCheckInterrupt()这一行，等待前面节点唤醒自己。

阻塞线程使用的是Unsafe的park方法阻塞线程。对应的解除阻塞的方法是unpark。使用park/unpark比wait/notify要好很多，因为notify是随机唤醒，notifyAll是全部唤醒，而unpark可以精确唤醒某一个线程，并且park并不需要像wait一样先获取synchronized锁。

  • 为什么要判断前驱是不是头节点再获取锁？
    因为头节点是持有锁的线程，它释放了锁的时候会唤醒后面的第一个非取消状态的线程（释放锁接下来会说），也就是说之后这个线程才该获取锁，所以只有前驱是头节点的时候才需要获取。
  • 为什么使用一个for死循环？
    前一个问题说了，头节点释放锁之后会唤醒第一个非取消状态的线程。但是锁是非公平锁，所以可能在这个时间点有新的线程调用了lock方式获取到了锁，所以被唤醒线程获取就会失败，就需要再次阻塞。

公平锁

通过构造函数可以指定创建的锁是公平锁还是非公平锁。
这里有一个误区是我之前遇到的，我以为非公平锁就是说持有锁的线程释放锁之后，所有被阻塞的线程都有相等的概率去获取锁，但是事实不是这样的，至少AQS中不是。AQS中限制的只是在持有锁的线程释放的时候队列中第一个被阻塞的节点和未加入到队列的线程可以同时去抢夺锁（unlock方法会唤醒持有锁线程节点后续的第一个非取消节点），但是之前已经被加入到队列并且阻塞的线程是无法去抢夺锁的，因为上面加锁的代码acquireQueued方法中有这样一个判断条件if (p == head && tryAcquire(arg))，只有头节点的后继节点才可以获取锁。

释放锁（unlock方法）

释放锁流程为：

  • 将锁的占用次数减1，看看是不是为0了，不为0代表没释放完（没有执行相同次数的lock和unlock）。
  • 如果为0了代表自己释放完了锁，则需要唤醒后继的一个线程，但是如果后继节点是被取消了或者数据异常节点为null，则需要找到离自己最近的为被取消的线程进行唤醒。
  • 唤醒之后就会回到acquireQueued方法，在下一次for循环调用shouldParkAfterFailedAcquire删除掉与头节点直接的取消节点，之后在下一次for循环中尝试获取锁。
  • 之后就是重复上述流程了。

共享锁

同样共享锁的获取AQS也是没有提供的，都是需要实际的类自己实现的，这里使用的代码是CountDownLatch的await方法，是一种共享锁的实现。
CountDownLatch是一个让一组线程等待一组线程完成的工具类，通过设定阙值，每一次有线程countDown方法这个计数就会减一，而计数为0之前调用await的线程会被阻塞。

获取锁

获取锁的流程为：

  • 线程先看计数state是否为0，为0表示countDown都调用完了，计数为0了，则表示自己获取了共享锁不需要阻塞。
  • 如果不为0代表需要被阻塞，那么就获取锁失败。
  • 锁获取失败的情况，调用doAcquireSharedInterruptibly方法，先使用addWaiter将自己插入到队尾，之后与独占锁的阻塞策略一样，通过shouldParkAfterFailedAcquire方法CAS将前驱节点的状态改成SIGNAL从而来表示线程需要被阻塞，之后parkAndCheckInterrupt通过CAS的线程阻塞方法将线程阻塞。表现在程序上就是调用await的线程被阻塞了。

释放锁

释放锁的流程为：

  • CAS+自旋将state减1，之后判断是不是减到0了，如果减到0了，则需要唤醒所有被await阻塞的线程。
  • 之后调用Unsafe唤醒方法唤醒后继的节点。
  • 唤醒后继之后，被唤醒的线程会回到doAcquireSharedInterruptibly方法中被阻塞位置，通过下一次循环调用tryAcquireShared获取到锁，之后调用setHeadAndPropagate方法，在方法中调用doReleaseShared()唤醒后继线程，之后被唤醒的线程会重复这步操作，一个一个的唤醒后续的线程直到队列中的线程都被唤醒。

总结

  • AQS是一个架子，它实现了阻塞队列的如插入节点、共享模式的节点唤醒方法、独占模式的节点唤醒等基于队列的操作，而暴露出现给继承者实现的是锁的获取逻辑，给继承者实现自己的特性。
  • 独占锁：就是将线程封装成一个Node插入到队尾通过Unsafe类的阻塞方法阻塞线程，之后头节点释放锁之后唤醒最近的一个为被取消排队的节点，这个节点去争抢锁，之后变为新的头节点。
  • 共享锁：与独占锁一样，将线程封装成Node插入到队尾，但是不一样的是唤醒策略，唤醒时是每个节点依次唤醒自己后的节点，直到全部唤醒。
  • 非公平锁：持有锁的线程释放锁后，队列中后继第一个线程可以与未加入队列的线程同时争抢锁。但是锁释放之前就已经在队列中的其他节点将还是阻塞的。
  • 公平锁：通过在获取锁的方法添加判断队列是否为空来让线程选择入队还是抢锁，公平锁需要让大家按照申请锁的顺序去获取锁，所以如果队列不为空就会获取锁失败而入队。
  • 整体流程：线程们cas修改state表示争抢锁，失败的进入队列中，通过判断前驱节点的状态是不是singal来判断自己是否需要被阻塞，当线程释放锁的时候会唤醒该节点的一个非取消状态的后继节点，如果是公平锁则可以获取锁，如果非公平锁就要和所有未入队的线程争抢锁。
