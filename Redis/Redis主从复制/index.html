Redis主从复制

本文介绍了Redis主从复制的原理

为了解决单点问题，实现故障恢复、数据备份、负载均衡，所以需要多个数据相同的redis节点，在节点挂掉的时候，其他节点拥有相同的数据可以服务或者实现多个节点分摊负载的功能，就出现了redis主从复制，这是解决redis高可用的基础。

主从复制

在节点上执行slaveof命令可以将节点指定为某个节点的从节点，slaveof同样可以取消某个节点的主从关系。

指定主从关系

slaveof host port
slaveof，从节点保存了ip和port之后这个命令就会返回，之后从节点会去建立与主节点的连接、复制主节点的数据等操作。

取消主从关系

slaveof no one
在从节点上执行这个命令，从节点会取消与主节点之间的连接，不会继续从主节点上复制数据，但是并不会删除掉自己节点的数据，只是与主节点的复制关系取消了。
但是如果执行的是切还自己的主节点操作的话，就会先断开与老主节点的连接，建立与新主节点的连接，删除自己的数据，复制新主节点的数据。

主从复制原理

主从复制流程

  • 保存主节点的信息
    首先在从节点上执行slaveof host port命令，从节点会保存host和port信息之后命令返回。

  • 从节点定时任务建立与主节点的连接
    从节点上的会执行一个每秒一次的定时任务，这个定时任务会拿着保存的ip和port尝试与主节点建立tcp连接。如果连接失败则会每秒一次尝试连接直到成功或者执行了slaveof no one命令。当建立了连接后，从节点会向主节点发送一个ping，如果主节点没有回复或者超时则从节点会断开与主节点的连接，之后每秒继续重复之前的操作。

  • 校验主节点权限
    当ping命令收到回复后，则代表连接建立并且可用了，则会校验master的权限配置与从节点是否相同。

  • 主节点发送数据给从节点
    之后主节点会触发一次bgsave，生成rdb文件，之后将这个rdb文件发送给从节点。这是一次全量复制，如果数据很多，则会很慢。

  • 从节点删除自身数据并加载主节点发送的数据文件
    从节点收到之后会清空自身的数据，之后加载rdb文件。加载完毕后，如果从节点开启了aof持久化则会立刻进行一次aof的重写操作，保存一份aof文件。

  • 持续维持数据复制
    后续主节点的写命令都会同步到从节点保持两端的数据一致性。

复制原理

复制分为全量复制和部分复制。全亮复制会生成rdb文件，可能会过大导致网络阻塞。部分复制是对全量复制的优化，防止每次都传输那么大的文件。

全量复制

前面说的复制流程其实就是一次全量复制的流程，从节点连接上主机点之后就会进行一次全量复制，将所有的数据拉过来。
1、主节点通过bgsave生成一个rdb文件，之后将rdb文件发送给从节点。
2、从节点收到rdb文件后，会清空自己的数据，之后加载rdb文件。
3、rdb文件加载完成后，如果开启了aof持久化就会立刻进行aof文件的生成。

部分复制

部分复制需要三个组件：

  • 节点运行id
    当redis节点启动后就会生成运行id，作为自己的唯一标识，当重启后就会生成新的id。

  • 复制积压缓冲区
    这个缓冲区会存储主节点的写命令，用于因为网络问题导致的主从数据复制不完整，通过这个缓冲区进行数据的修复。

  • 复制偏移量
    主从节点都会存储这个复制偏移量，当主节点进行写操作时，会给偏移量追加命令占用字节数，从节点接收到数据，也会追加偏移量。部分复制时从节点会发送自己的复制偏移量和自己主节点的运行id。主节点收到命令后，会判断id和自己相不相同，之后根据主从各自的偏移量，从复制积压缓冲区中把命令同步给从节点。

这样可以防止大大减少复制时传输的数据大小。

异步复制

复制的过程是一个异步的操作，也就是说主节点发生写命令之后会直接将命令发送给从节点，主节点处理完写命令直接就返回了，并不会等待同步完成。这样就会造成主从之间的数据延迟，主节点写完数据，从节点可能读到的还是老数据。

主从结构

一主一从

一主一从是简单的结构，可以关闭主节点的持久化，在从节点上进行持久化。这种结构支持主节点宕机从节点进行顶替的情况，但是当主节点宕机之后不要进行主节点的重启，因为主节点一旦重启，主节点并没有持久化数据，所以主节点是空的，那么通过主从复制会将从节点的数据也清空。
这种时候可以在从节点上执行salveof no one去掉主节点。

一主多从

这种情况有多个从节点，数据备份比较多，而且可以做读写分离，但是会对主节点造成压力，因为要同步多个从节点。

树状主从

这种结构是一个主节点，下面有从节点，从节点又是其他节点的主节点。这样的结构可以减轻主节点同步从节点的压力。

其他

读写分离

这种结构是读取操作请求都打到从节点上，写的操作都放到主节点上执行，这样可以减轻主节点的压力，但是会有主从复制引起的数据延迟。

什么时候会全量复制

全量复制会将主节点所有的数据生成rdb文件发送给从节点，这会非常消耗资源，阻塞网络。

  • 当从节点第一次连接上主节点：因为从节点没有数据，必须将主节点的数据同步过来。
  • 当运行id与主节点的不匹配的时候：这说明主节点重启了，或者从节点切换了主节点，则必须清空从节点进行一次全量的复制保证数据一致。
  • 当主节点的复制积压缓冲区溢出时：因为缓冲区移除了，则无法将完整的数据补偿给从节点，则必须进行一次全量复制以保证主从节点数据一致性。
