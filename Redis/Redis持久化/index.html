Redis持久化

本文Redis中aof、rdb两种持久化的原理，以及它们的区别。

Redis持久化是为了防止内存中的数据在Redis重启后丢失，所以生成了一个持久化文件，用来保存在内存中的数据，当redis重启之后通过持久化文件恢复内存中的数据。
Redis提供了两种持久化方式，分别是RDB和AOF。

RDB

RDB是如何进行的

RDB方式是通过生成redis内存快照文件的方式来进行持久化的，rdb文件是一个二进制文件。一共有两个命令，分别是save个bgsave。save命令阻塞住redis进程直到完成持久化，如果这个时候redis中的数据很多的话，就会造成后边的命令无法执行。所以出现了bgsave命令，这个命令会使得redis进程fork出一个子进程，而主进程只会阻塞到子进程fork完毕就会结束阻塞，随后通过这个子进程完成redis的持久化操作，bgsave的流程如下：

  • redis接收到bgsave命令，检查当前是否有正在持久化的AOF和RDB进程，如果有的话就直接返回了。
  • 否则话redis主线程会阻塞进行子进程的fork操作，当子进程fork结束之后，子进程返回成功。
  • 主线程停止阻塞，继续处理队列中的后续任务。
  • 子进程生成redis的内存数据快照，之后原子性替换现有的rdb文件。
  • 子进程结束之后通知主线程持久化结束，主线程更新自己相关信息。

手动/自动

RDB持久化方式分为手动执行和自动执行，手动执行就是save和bgsave命令，但是在以下情况也会触发自动执行：

  • 配置文件的save m n（redis在m秒内如果出现了n次修改，就自动进行bgsave）
  • 从节点进行全量复制时，主节点会触发bgsave，之后将文件同步给从节点。
  • debug reload时会触发bgsave
  • 在进行shutdown命令时，如果没开启aof也会自动bgsave。

优缺点

  • RBD是二进制文件，在redis恢复时使用rbd文件比aof快很多。但是rbd不能做到实时的持久化，所以可能会丢失数据。

可以通过redis配置开启rdb文件压缩，可以节省内存，而且在主节点和从节点之间同步rdb文件时可以减少网络传输。

AOF

AOF是如何进行持久化的

所有的写命令都会先追加到缓冲区中，当到达了redis配置的刷盘时机，就会触发aof缓冲区刷入aof文件的同步磁盘操作。

aof文件写入的三种时机

  • 每次写入都进行刷盘：会大大降低Redis的性能，因为每次写操作都要同步磁盘，但是安全性最高，不会丢失数据。
  • 每秒进行一次刷盘：每一秒都会执行fsync命令将缓冲区数据追加到aof文件中，但是如果这时宕机了就会丢失一秒钟的数据。
  • 不刷盘：不主动刷盘。通过操作系统自动进行刷盘，可能会丢很多数据但是效率高。

AOF文件重写

aof文件会越来越大，这时当大小到达了阙值就会触发aof文件重写的操作，通过合并命令和删除无效命令来减小aof文件，流程如下：

  • redis主进程fork出一个子进程，这个期间redis是阻塞的，子进程fork出来之后主进程解除阻塞。
  • 子进程将文件进行重写，这个过程中主进程继续进行正常的刷盘操作，但是新的命令还会存储到另一个缓冲区（aof重写缓冲区）
  • 当子进程结束重写后会通知主进程，之后主进程将aof重写缓冲区数据刷入aof文件，之后完成重写。

AOF存储

AOF中保存的命令是文本协议的，当重新加载时通过重新执行aof文件的命令来恢复数据，所以重写也可以减少重新加载的时间。
AOF保存文本协议是一方面是为了可读性，一方面不用进行命令的转化造成二次开销。
AOF缓冲区的作用是防止每次写入都进行刷盘，这样redis的性能就会大幅度受到磁盘io的影响。

Redis加载持久化文件过程

  • 1、先判断是否开启了aof持久化，如果没有开启走步骤2，否则走3。
  • 2、检查是否开启rdb持久化，如果开启了，则进行rdb的数据加载。
  • 3、通过aof文件进行数据加载。

RDB和AOF对比

  • RDB文件存储的是二进制数据，aof存储的文本数据。
  • rdb是通过复制内存快照方式持久化，aof是将执行的写命令保存到aof文件。
  • rdb不适合实时持久化，aof适合实时持久化。
  • rdb的数据恢复速度要大于aof。
