Redis主从模式就是为了提高Redis的可用性，但是正常的Redis主从模式无法进行故障的及时发现和自动的故障转移等操作，为了保证高可用就出现了Redis Sentinel。

为什么需要Redis Sentinel

一个主从结构的Redis服务，当Master故障之后都要做什么？
1、要挑选出一个slave节点手动设置为master节点。
2、改变其他slave节点的master指向。
3、通知客户端master节点的信息变化。

当前主从结构的Redis的整个故障转移过程都需要人工干预，而且并不能及时的发现主节点故障。为了解决这些问题保证redis的高可用，就出现了redis sentinel。redis sentinel可以监控每个redis的状态并及时发现故障，自动进行故障转移、保存节点信息、通知客户端。

Redis Sentinel原理

三个定时任务

Sentinel只需要主节点的信息就可以开始工作了，原因是因为有下面的三个定时任务。

  • 每10秒给主节点发送info命令，主机点会回复自己的信息，包括从节点的信息，这样sentinel就可以获得整个主从redis结构的拓扑结构，也可以及时发现新加入的从节点。
  • 每2秒给master节点的_sentine_:hello频道发送自己对主节点的判断和自己的信息。所有的sentienl节点都会订阅这个频道，所以sentinel发送信息之后，其他的节点都会收到这个消息，他们就彼此知道了对方的信息，就可以进行互相的连通。
  • 每1秒sentinel给master节点、slave节点、其他sentinel节点发送ping命令来检测是否存活，因为前两个定时任务让sentinel节点知道了这个架构中其他节点的信息，所以就可以进行连接并且进行心跳检测。

主观下线、客观下线

  • 主观下线
    每一秒Sentinel节点都会向其他的节点发送心跳命令，如果超时了还没有获得响应，则会认为目标节点下线了，并且会将目标节点标记为主观下线状态。

  • 客观下线
    当sentinel节点主观下线了Master节点，他会给其他的sentinel节点发送sentinel is-master-down-by-addr命令，响应就是目标sentinel是否也将Master节点主观下线了，如果当前sentinel收集到了足够的主观下线意见，那么master就会被标记为客观下线。

sentinel is-master-down-by-addr命令通过参数的不同可以表示为两种请求。一种是‘*’参数，一总是携带自己runid的‘runid’参数。‘*’参数的时候代表询问目标节点对于master节点的主观下线意见，而‘runid’的命令表示让目标节点给自己投票。

故障转移流程

选举领导者Sentinel节点

当sentinel对master进行了客观下线后，那么需要挑选出一个sentinel节点进行master节点的故障转移操作。
sentinel领导者的选择是通过Raft算法实现的。流程如下：
1、将master客观下线之后，这个sentinel节点向其他的sentinel节点发送sentinel is-master-down-by-addr命令，携带自己的runid参数，表示需要别的节点对自己投票。
2、每个节点都只有一次投票的机会，它会将票投给第一个给自己发送命令的人，后续的投票请求将会被忽略。
3、当sentinel节点收到了max（quorum，(sentinel节点数/2)+1）的票数后，就会成为领导者。quorum是客观下线master所需的主观下线数。
4、如果没有选出领导者会进行下一轮的选举。

转移过程

1、首先这个领导者sentinel会按照规则在slave节点中挑选出一个节点，规则如下，按照从上到下的顺序选择：

  • 节点优先级配置最高
  • 复制偏移量最大（表示与master的数据最相近）
  • runid最小

2、选择出来之后，会对其进行slave of no one，之后将其他的slave的master指向它，之后将新的master的消息发布，客户端就可以根据订阅的频道感知到新的master。
