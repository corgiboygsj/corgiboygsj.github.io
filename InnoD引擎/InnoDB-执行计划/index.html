什么是执行计划

一条sql会经过mysql优化器的优化后变成一个所谓的执行计划，这个执行计划可以看出来这条sql是如何被mysql来使用的。explain可以帮助我们来查看执行计划，从而我们可以通过执行计划对sql进行优化来提升性能。

explain各个列的含义

建立一张表来举例

table

每一条sql无论联查了多少个表，到最后都需要对每个表进行单表访问，explain列出的每一条数据代表对一个单表的访问方式信息。

type

这个列描述的是表的访问类型，也就是访问方式，比如是不是走索引，怎么走的索引等。

const

当通过主键或者唯一索引进行等值搜索时就是const。

eq_ref

当连表查询并且是通过主键或者唯一索引进行相连的话才会使用eq_ref。

ref

当通过通过二级索引进行等值匹配，查询的方式就是ref。

ref_or_null

通过二级索引列查询，匹配时指定等于null就会走ref_or_null

index_merge

这种情况是一个查询使用到了多个索引，一般出现在or取交集、and取并集这种情况，where条件对应多个索引，就可能会发生这种情况，最后将结果进行union操作。
这种情况效率不高，这种情况可以考虑建立联合索引

unique_subquery

当mysql要将in查询转化为exists时，子查询如果可以走主键或者唯一索引，那么子查询就是unique_subquery类型。

index_subquery

与上面相同，只不过子查询走的是二级索引。

in和exists区别（有子查询情况）

  • in的查询过程：先执行子查询，将结果拿到外层查询中进行查询，实则是一个范围查询。
  • exists查询过程：实则是一个循环的过程，算法叫做Block Nested-Loop Join，这个算法通过将外部表的分成组一部分一部分放入join buffer中，之后进行循环将buffer中的记录与子查询结果集的记录进行比较，这样可以减少循环次数。

过程虽然相似，但是不同的是in的内外查询都可以走索引，而exists只有子查询可以走索引。但是这也不能代表exists的效率小于in，在外部表结果集大于子查询结果集的情况下一般in的查询效率更高，相反外部小于子查询结果集合情况exists的效率可能会更高。

range

范围查询会走range，比如<、>、between and。

index

使用了二级索引进行全表查询，比如使用到了覆盖，where条件没有走索引，但是where条件和查询的列都在二级索引中，就会扫描整个二级索引查询数据。

All

与index差不多，但是扫描的是聚簇索引，并且进行了整个索引的扫描，因为二级索引列更加简洁，而聚簇索引包含所有的列还有隐藏列，所以index的性能要大于all的性能。

type的效率从上到下依次降低。

id

一个select就对应有一个id，但是有时候mysql会计算成本，将子查询变成连接查询，也就是说最后只会产生一个id。通过看有几个id可以知道mysql是如何进行优化的。

还有一种其他的情况。

可以看到多了个第三行的id为null，原因是union需要对结果进行去重，因为要去重就需要将两个查询的数据放到一起，就需要建立一个临时表，也就是第三条记录的 <union1,2> 表，在这个表中再进行处理，因此有了第三条记录。
但是union all就不会产生第三条记录，因为union all是将两个查询记录合并而不去重。

select_type

select_type表示单表的查询方式。

simple

查询中不包含union和子查询的都是simple类型。连接查询两个表也都是simple的。

primary和union

包含union、union all的查询，最前面的查询是primary，其他的查询都是union的。

union_result

union两个表查询，生成的临时表的select_type就是union_result。

subquery

如果子查询不能优化为join查询的话，而且子查询是不相关子查询的话，那么子查询就是subquery类型。

dependment subquery

如果子查询不能优化为join查询的话，而且子查询是相关子查询的话，那么子查询就是dependment subquery。

##possible_keys和key
possible_keys是可能会使用到的索引，这个列的值越多，则mysql需要进行计算成本的时间就越多。
key是查询时候实际使用到的索引。

key_len

这个代表查询所使用到的索引列的字节长度。

ref

表示的是走索引访问的时候，是用什么进行匹配的。

可以看到就是通过常量匹配key1=a，const代表的就是常量，type为ref代表走的索引进行的等值匹配。

不通过常量匹配时，通过连表两个列连接时，可以看到使用的就是连接的列在索引中进行的查找。

Extra

用来描述额外的信息。

No tables used

当查询没有from时会是No tables used。

No matching min/max row

当使用min或者max函数时，没有匹配的行时会是No matching min/max row。

using index

当使用覆盖索引的情况下，就会using index。

using index condition

使用到了索引，但是不能全都走索引的时候会是using index condition。

key1>'z’走了索引，但是key1 like '%a’不能走索引了，这时筛选出key1>'z’的数据之后，并不是立刻回表查询之后再筛选出key1 like '%a’的数据，而是直接从key1的索引中筛选完之后，再筛选出key1 like '%a’的数据，之后再回表。这样可以降低回表的数据量。

Using where

当查询包含索引列查询还包含非索引列查询的情况下，就会是using where。

Using join buffer (Block Nested Loop)

当被驱动表不能有效利用索引时，就会使用Using join buffer (Block Nested Loop)。
通过分配一块join buffer空间，将驱动表的数据放进去，之后去循环访问被驱动表。

Using filesort

当需要对结果排序的时候，而且不能排序列不能通过索引的话，就会是filesort。
