什么是事务

事务可以将对于数据库的一系列操作绑定到一起，让这些操作成为一个整体，一起成功或者一起失败。

事务的四个特性

原子性

一系列操作要么都成功、要么都失败。数据会变成操作之前或者整体操作之后的状态。

一致性

事务提交提交/回滚后数据的完整性不会受到破坏，比如转钱一个人扣钱一个人加钱，但是最终的数据的状态是正确的，不只是说一个人扣了一个人加了就完事了，还要保证双方的钱都是大于0的。就比如秒杀系统中不要超卖一样，数据必须是正确的。因为一致性比较抽象我个人也不是特别清楚，所以引用了知乎上一段回答。

一致性很抽象，它不只是需要数据库来保证，还需要应用程序来保证，其他三个特性保证的是数据库内部的数据一致性，但是跟业务逻辑挂钩的一致性需要程序员编写程序进行保证。

隔离性

表示两个事务之间的操作互相不干扰。对于一个事务来说，其他的事务它是看不到的，就好像只有它自己在进行操作。InnoDB提供了四种隔离级别，来限制事务之间的隔离程度，对应级别的并发性能也是不同的。

持久性

已经提交并且返回成功的事务对数据的影响会永久的保存到磁盘中，无论数据库或是服务器宕机，当数据库重新启动时都会恢复到提交之后的状态。

持久性实现

持久性主要是通过Redo Log实现。每一次对数据的修改都会被记录到Redo Log中，Redo Log是一种物理逻辑日志，它记录了被修改的数据页号、记录在页面中的偏移量、被修改的字段修改后的值。因此即使数据库宕机，只要日志在就可以通过日志恢复宕机之前数据库的状态。

InnoDB采用日志先行原则，也就是说我们在操作完数据页并且提交事务返回成功后，被更改的数据页并不一定会落到磁盘上。但在事务返回成功之前会先把Redo Log Buffer中的Redo Log都写到磁盘中，之后才会返回成功。因为写入Redo Log日志是一个顺序写的过程，所以相比将事务影响的数据页刷盘带来的随机读写磁盘效率高很多。

故障恢复

在数据库启动的时候总会去加载redo log进行数据的恢复。恢复过程分为两个阶段：

  • 第一阶段：去redo log file的头部看两个checkpoint的checkpoint no哪个更大，之后获取较大的checkpoint的lsn和offset，定位到redo log文件Redo Log的位置，从这里往后读取redo log并且重放。
  • 第二阶段：因为redo log中可能存在未提交事务的redo log也被刷入磁盘中，而且在提交时还会出现bin log写入成功，redo log提交失败的情况，这种会造成主从之间的数据不同步。因此这个阶段借助bin log和undo log来对这种数据进行修复。首先从bin log中获取log对应的事务id列表xids（bin log里的事务都是已经提交了的事务），从undo log中获取未提交事务的列表。未提交事务列表中id存在于xids中的都需要提交，其他的全部回滚。这样就把那些未提交事务却刷盘的redo log给恢复到原本的状态了。

原子性

事务一共有两个动作，一个是commit、另外一个就是rollback。其中commit就是上面说的持久性进行的工作。而负责rollback的是一个名为undo log的日志，它是一种逻辑日志。它保留了被修改数据之前的版本，可以认为是数据之前的一个快照，再执行rollback时可以从放这个undo将数据恢复。

隔离性

重点就是这个隔离性，其中涉及到的东西非常，很容易懵逼。接下来的文章会先说明InnoDB提供的四种隔离级别以及他们的区别，还会说明他们是如何实现的。其中需要注意的是要把非锁定读和锁定读分开来看，多版本并发控制（MVCC）跟锁无关，锁定读才跟锁有关。锁和MVCC一起实现了隔离级别，另外所有的写操作都是一种锁定读。记住这些再往下看。

并发访问下出现的问题

  • 更新丢失：两个事务分别对数据查询后的结果进行修改，后提交事务将先提交事务的修改结果覆盖的问题。
  • 脏读：事务读取到其他事物修改但未提交的值，比如A将数据从1改成2，B去查出等于2，之后A回滚，B读取的值就是没有意义的。
  • 不可重复读：没有脏读的问题，但是会因为其他事务提交而出现两次读取相同数据的值不同的情况。A读取值为1，之后B修改成2并且提交，A再读取发现值为2。
  • 幻读：两次相同查询的结果集的大小不同。A查询一次获取到3行，B在A的查询结果集中删除或者添加1行数据，之后A再次查询发现两次结果集不同。

事务的四种隔离界别

隔离级别是用于限制事务和事务之间对于数据的可见性大小的，它分别解决了上面的不同问题，越高的级别解决的问题越多，但是性能也会更低。

  • 未提交读：会读取到别的事务更改后未提交的数据。这种会出现脏读的现象。
  • 已提交读：解决了脏读的问题，但是会读取到其他事务提交之后的数据。也就是说会出现不可重复读的问题。
  • 可重复读：解决了脏读、不可重复读的问题。这种情况会出现幻读，但是InnoDB的可重复读不会出现幻读情况，它在非锁定读的时候通过MVCC避免，在锁定读的时候通过Next-Key Lock进行避免。
  • 序列化：解决了脏读、不可重复读、幻读的问题，但是并发度最低，因为该隔离级别下，InnoDB会给每一个读操作都隐式的加上一个共享锁(lock in share mode)，从而读写操作彼此互斥达到事务顺序执行。

下面的知识一定要把每个小节结合到一起来看。非锁定读和MVCC结合到一起。锁定读和锁结合到一起来看，才能方便理解。

非锁定读/锁定读

在理解实现之前要先明白锁定读和非锁定读的区别，才能好理解后面的东西。
非锁定读还被称为快照读，锁定读被称为当前读。

非锁定读

正常的select操作，比如select … from … where …;这样的都是非锁定读。

锁定读

  • 所有的写操作insert、update、delete都是锁定读。
  • 加锁的select方式。select … from … where … for update/lock in share mode;

这些操作会给被影响的数据加锁，具体怎么加行锁还是表锁，后面讲锁时候会说。

多版本并发控制（MVCC）

快照读就是通过多版本并发控制实现的，多版本并发控制又被称为MVCC。它只作用于已提交读/可重复读这两个隔离级别之下，并且只在非锁定读下起作用。
先简单说下多版本控制的大致流程。MVCC大致的思路是给数据行建立一个版本的概念。每次修改数据都会将原数据记录下来，并且与当前数据版本连接起来形成一个版本链。当事务读取数据时，会根据当前事务的隔离级别在不同时机生成Read View。之后从版本链头部开始遍历（最新版本开始），拿着数据的版本按照一定规则和ReadView中的属性进行比对，如果满足条件那么该版本的数据就是该事务可见的。

写数据也可以理解成原数据不动，而是在链表头部新插入一个节点，并且原数据连接起来。

先看一下MVCC中用到的一些东西：数据行的隐藏列、Read View、版本读取规则。

数据行的隐藏列

InnoDB会给每一个数据行添加隐藏列，一共有三个。

  • TRX_ID：产生这条数据的事务id。
  • ROW_ID：数据的行ID，只在没有主键的时候才会添加。
  • ROLLBACK_POINTER：回滚指针，用于指向该行数据的上一个版本的指针。

Read View

ReadView中属性的值依赖于当前系统中活跃的事务，版本可见性比较的就是ReadView中的属性和版本链数据的trx_id。
已提交读级别会在每一次执行select的时候生成ReadView，而可重复读级别只会在事务第一次执行select的时候生成ReadView，在事务结束之前不会改变了。
已提交读之所以会出现不可重复读的问题，就是因为他每次select都生成新的ReadView，所以可以看到两次select之间提交的事务的修改结果。

Read View的结构如下:

  • m_ids：当前系统中存活的事务id的列表。
  • min_trx_id：当前系统中存活事务的最小事务id，就是m_ids中的最小值。
  • max_trx_id：系统下一个即将分配的事务id，注意这个不是m_ids中的最大值，因为比m_ids最大值还大的事务可能已经提交了。
  • creator_trx_id：当前事务ID；

版本读取规则

使用Read View读取数据时的规则如下，满足规则的版本才是事务可见的。比较成功后就会返回数据，版本链后续数据就不会再比较了。

  • min_trx_id大于等于行trx_id，该行可读，否则拿下个版本比较。
  • m_ids中包含行trx_id，该行不可读，因为该行的事务未提交。注意是m_ids包含trx_id，而不是trx_id在集合最大最小事务id的范围之内，trx_id必须是存在于m_ids集合内的元素。
  • 行trx_id位于m_ids最大值和max_trx_id之间，行可读，因为事务已提交。
  • 行trx_id大于等于max_trx_id，行不可读。

另外事务id是在事务第一次做写操作的时候才会生成，而不是开启事务时生成。

锁

非锁定读是通过读取快照返回数据的。而当前读（锁定读）是通过给数据加锁实现的，当前读读取的都是当前数据最新的数据，不会读到之前版本的数据。

InnoDB提供了下面的锁

  • table lock（表锁）当写数据的时候，如果条件不能使用到索引就会将整个表的数据都锁住，即使其他事务要修改之前事务没有改动的数据也会被阻塞。
  • row lock（行锁）当写数据时走了索引，那么就会只会给被影响到的数据行加锁。此时其他事务要操作其他数据是不会被阻塞的，只有他们操作的数据重叠才会阻塞。
  • gap lock（间隙锁）锁定两个数据行的间隙，比如有两条记录1，5。gap lock锁住的是2,3,4这些数据，可以保证其他事务无法在本事务释放锁之前在1-5之间插入数据。
  • next-key lock（行锁+间隙锁）行锁和间隙锁的组合。1，5的数据，锁的就是1,2,3,4,5这几个数据。

在InnoDB中锁定读的幻读就是通过Next-Key Lock解决的，它通过锁住行和区间，让其他事务在这个区间内的操作阻塞，直到事务提交释放锁。

提一嘴需要注意的，写锁与读锁、写锁互斥，读锁与读锁不互斥。行锁的不是数据行而是索引行，比如通过二级索引修改，那么会锁住二级索引的行之后再锁对应行的聚簇索引的行，最终锁定到的是聚簇索引。
